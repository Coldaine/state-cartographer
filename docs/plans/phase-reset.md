### What went wrong

The project thesis is: "build state graphs from observation." What actually
happened was someone read `page.py`, read `assets.py`, and wrote
`alas_converter.py` to transpile Python class definitions into JSON. The
"graph" is a transcription of ALAS's source code, not a product of observing
a running system.

- No screenshot was ever classified successfully
- No anchor was validated against a real frame
- No transition was confirmed by watching it happen
- The RGB values are ALAS region-average colors copy-pasted from `Button.color`
  tuples, not sampled from actual pixels

Then the project jumped straight from "transpile ALAS" to "build
executor/scheduler/runtime" -- skipping the entire observe-calibrate-validate
loop that IS the project's reason for existence.

The runtime modules (scheduler, resource_model, task_model) are declarations
of intent shaped like code. They have tests, but the tests only prove the
stubs work against other stubs.

### What's actually salvageable

1. **pilot_bridge.py** -- DroidCast screenshot capture + ADB tap/swipe. Works.
2. **~2000 real screenshots** in `data/` -- captured but never consumed.
3. **calibrate.py** -- the code is correct, just never fed real data.
4. **locate.py** -- the logic is sound, just operating on garbage anchors.
5. **alas_log_parser.py** -- can parse ALAS logs and extract page transitions.
6. **The ALAS vendor code** -- a reference implementation with 4 screencap and
   4 touch methods, battle-tested against the actual game.

## The Correct Approach

### Principle

ALAS is a working automation system. It takes screenshots, classifies states,
taps buttons, and navigates. Every single one of those actions is observable.

**The correct methodology:**

1. Run ALAS normally for an extended session
2. Sit alongside its screenshot/action pipeline (sidecar, not monkeypatch)
3. Record every `(screenshot, page_label, task_label)` tuple --
   where `page_label` comes from ALAS's own log output, not from our broken classifier
4. Feed that labeled corpus into `calibrate.py` to learn real pixel RGB values
5. Validate `locate.py` against held-out screenshots from the corpus
6. Let the state machine and its anchors emerge from observed data, not from
   reading Python source

That's the entire thesis of the project: "build state graphs from observation."

### Architecture: The Sidecar

```
Terminal 1: ALAS (user launches)       Terminal 2: Sidecar (agent launches)
  |                                      |
  |-- logs "Page switch: X -> Y"         |-- tails ALAS log file
  |-- logs "Start task: Commission"      |-- tracks current page + task
  |-- runs for hours                     |-- captures screenshots via PilotBridge
  |                                      |-- writes labeled JSONL index
  |                                      |-- on exit: prints corpus summary
```

The sidecar does NOT call `locate()`. It does NOT try to classify screenshots.
It uses ALAS's OWN log output as ground truth labels. That's the whole point --
ALAS already knows what page it's on. We just record its answers alongside our
screenshots.

### What the sidecar produces

```
data/corpus/
  index.jsonl           # one record per capture
  000001.png            # screenshot at capture time
  000002.png
  ...
```

Each JSONL record:
```json
{
  "ts": "2026-03-20T14:23:01Z",
  "path": "data/corpus/000001.png",
  "alas_page": "page_main",
  "alas_task": "Commission",
  "log_line": 4821
}
```

### What happens after the sidecar run

1. **calibrate_from_corpus.py** -- groups screenshots by `alas_page`, samples
   pixel RGB at grid coordinates across each group, finds pixels that are
   stable within a page and discriminating across pages. Writes real anchors
   to `graph.json`.

2. **validate** -- runs `locate()` against held-out screenshots from the corpus.
   Reports match rate per state. Identifies anchors that need more data or
   different coordinates.

3. **Iterate** -- run more ALAS sessions, capture more data, refine anchors,
   until `locate()` achieves >90% match rate on real game frames.

4. **THEN build runtime** -- once the graph is grounded in reality, the
   executor/scheduler/daemon code has something real to drive against.

## Execution Order

### Step 1: Build `scripts/alas_sidecar.py` (NOW)

The observation tool. Must exist before anything else matters.

- Tail ALAS log file (poll-based, handles file not yet existing)
- Parse "Page switch: X -> Y" to track current page
- Parse "Start task: `X`" / "End task: `X`" to track current task  
- Capture screenshot every N seconds via PilotBridge (or ADB fallback)
- Write to `data/corpus/index.jsonl` + save screenshots
- Print summary on CTRL+C (N captures, M unique pages, top pages by count)

Dependencies: `alas_log_parser.py` (NavigationAnalyzer, TaskAnalyzer),
`pilot_bridge.py` (PilotBridge). No dependency on `locate.py`.

### Step 2: Run ALAS + sidecar for >=30 minutes (USER ACTION)

User launches ALAS in their terminal. Agent launches sidecar in a second
terminal. ALAS runs its normal task loop. Sidecar captures labeled screenshots.

Goal: >=200 labeled screenshots covering >=10 unique pages.

### Step 3: Build `scripts/calibrate_from_corpus.py`

Consumes the JSONL corpus. For each page:
- Load all screenshots labeled with that page
- Sample pixel RGB at candidate anchor coordinates
- Find coordinates where RGB is **stable** across all screenshots of the same
  page (low variance) and **discriminating** across different pages
- Write the winning (x, y, expected_rgb) anchors to `graph.json`
- Report: states calibrated, anchor stability scores, pages needing more data

### Step 4: Validate

```bash
uv run python scripts/locate.py --graph examples/azur-lane/graph.json \
  --screenshot data/corpus/000042.png
```

Run locate against every screenshot in the corpus. Compare location result
to the ALAS label. Report match rate. This MUST achieve >80% before
proceeding.

### Step 5: THEN resume runtime work

Only after the graph is grounded in observed data:
- Wire executor to use calibrated graph
- Build daemon loop
- Build emulator startup tool
- All the stuff in automation-runtime-consolidation.md
