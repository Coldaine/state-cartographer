Decisions and findings from this session

📍

MEmu is on 21513, not 21503

adb devices returned 127.0.0.1:21513 device. All configs say 21503. Fix: update configs/memu.json and transport/config.py defaults.

📸

ADB screencap works. Screen is currently black.

5 consecutive captures: 103–128ms, all 1280×720, valid PNG magic headers. Mean pixel value = 0.0 — game is not running or is minimized. The capture infrastructure is proven. The screen content is not game state yet.

🔍

NemuIpc is MuMu12 (Netease), NOT MEmu (MicroVirt)

ALAS's nemu_ipc.py loads external_renderer_ipc.dll, which ships with Netease's MuMu12 emulator. We use MEmu 9 — no equivalent native DLL exists. ADB screencap is the correct screenshot method for MEmu.

💀

MaaTouch last released March 2024 — 12 months ago

The binary at vendor/AzurLaneAutoScript/bin/MaaTouch/maatouchsync is from a frozen project. StarRailCopilot (LmeSzinc's next project, actively maintained as of March 2025) dropped MaaTouch entirely. Uses adbutils==1.2.9 + uiautomator2 instead.

⚡

adbutils 2.12.0 has native tap/swipe — no MaaTouch needed

Modern adbutils (Nov 2025): d.click(x, y), d.swipe(x1,y1,x2,y2,duration), d.send_keys() built in. This was not in 0.11.0. ALAS pinned to 0.11.0 because those methods didn't exist. We don't have that constraint.

🗑️

transport/discovery.py and transport/maamcp.py are dead weight

discovery.py auto-discovers MaaMCP (not installed), adbfriend (not a thing), scrcpy (debug only). maamcp.py calls from maa.controller import AdbController which fails every time. Both files exist but nothing works through them.

❓

Touch method: adbutils 2.x native vs uiautomator2 vs MaaTouch

Three options: (1) adbutils 2.12.0 built-in d.click() — simplest, no extra daemon; (2) uiautomator2 — matches StarRailCopilot architecture but requires device-side atx-agent; (3) MaaTouch socket — lower latency but unmaintained. Need decision before implementing maatouch.py.

❓

scrcpy 3.x programmatic frame access on Windows

scrcpy 3.0+ added virtual display with frame capture API — no longer Linux-only. If py-scrcpy-client supports scrcpy 3.x, this could replace ADB screencap for lower latency. Not yet evaluated with scrcpy 3.3.4 (our installed version). Currently classified DEBUG_ONLY per probe from earlier today.

❓

memu.json — what is it, what should it own?

Currently: pinned ADB serial + "tool posture" (which control method to use). Tracked in git. The serial it contains (21503) is WRONG (real is 21513). It defaults primary_control to "maamcp" everywhere. Proposal: make it own only the serial. Tool selection moves to code defaults.
